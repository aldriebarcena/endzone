## Open questions — all resolved (as far as they can be)
Every gap originally noted here has been either fully closed or converted into an honestly-documented, reasoned limitation. None are "haven't gotten to it yet."

- **~~No API surface for the iOS client to write `LeagueConfig`~~ — resolved.** `backend/template.yaml`'s `EndzoneHttpApi` (API Gateway HTTP API) exposes `POST /leagues` (`api/import_league/app.py`) and `PUT /device-token` (`api/register_device_token/app.py`). Auth is Sign in with Apple's identity token verified by API Gateway's native JWT authorizer against Apple's public JWKS (`issuer: https://appleid.apple.com`, `audience: <bundle ID>`) — no custom token-verification code needed in either Lambda; `requestContext.authorizer.jwt.claims.sub` is already a trusted `userId` by the time a handler runs. iOS side: `AuthManager` now captures `credential.identityToken` (was being discarded before — a real gap, fixed), `EndzoneAPI`'s `importLeague`/`registerDeviceToken` take an `authToken` param, and a new `LeagueImportView` (reachable via a toolbar button on `LiveFeedView`) makes this demoable — screenshot-verified rendering correctly in the simulator. Untestable end-to-end since the API isn't deployed (see scope decision) — verified by matching the backend's actual routes/handler code instead.
- **~~Device-token registration~~ — resolved**, same fix as above (`PUT /device-token`); `push.py`'s read side already existed.
- **~~Fantasy point computation not wired into poller/push~~ — resolved.** `poller.py` now fetches ESPN's enrichment data per new event (gracefully degrading to no enrichment if ESPN fails or `espn_game_id` is missing — Tank01 diffing remains the source of truth for "did a score happen" either way), matches it, and packages the raw ingredients (matched play, plus only the player categories/names relevant to that play) into the SQS message. `push.py` computes actual points **per subscriber**, using that subscriber's own `LeagueConfig.pointValues` — the same play is worth different amounts to different people, so this couldn't live in `poller.py` as a single canonical number. `checker.py` now passes `espn_game_id` through the Step Functions execution input (only available from `getNFLGamesForDate`, not the box-score endpoint `poller` calls) so no extra Tank01 request is needed to look it up.
  **Coverage extended with more real verification, done live during this pass:** queried ESPN's play-by-play for three more real games (beyond the original 6-play sample) and confirmed real `"Field Goal Good"` plays with `statYardage` matching actual kick distance (40/57/25 yards, cross-checked against each play's real text). Field goals are now scored using Sleeper's real distance-tiered rates (`fgm_0_19`/`fgm_20_29`/.../`fgm_50p`) — a different computation shape than touchdowns' yardage×rate model, so they get their own code path. **Deliberately still not mapped:** safeties and defensive/special-teams TDs (also confirmed-real ESPN play types, ids 20/36/39) — not a data gap but a genuinely undesigned scoring model (individual-defender vs. team-defense credit was never decided, and zero Tank01 sample data exists showing what category name, if any, Tank01 gives defensive players).
- **~~Box-score status strings beyond "Completed" unverified~~ — resolved, better than expected.** Tank01's box score already returns a numeric `gameStatusCode` (0 not started, 1 in progress, 2 final, 3 postponed, 4 suspended) alongside the free-text `gameStatus` — documented as consistent across every Tank01 endpoint (found during Phase 3 research, never wired in until now). `GameState.status_code` and `poller.py`'s `is_final` check now use this instead of fuzzy-matching text strings that weren't even consistent between Tank01's own endpoints (`"Completed"` from the box score vs. `"Final"` from the games list, for the same finished game).
- **~~Checker's "which game to track" is still a placeholder~~ — resolved as a reasoned dead end, not fixed.** Analyzed the actual cost of doing this properly: ranking live candidates by score needs a box-score call per candidate (Tank01's games list carries no score field) — a normal NFL Sunday (up to ~13 simultaneous live games across slates) could burn a meaningful fraction of the entire monthly budget in a single day just to rank candidates. And even fully solving that only gets to "highest-scoring globally," not DESIGN.md's actual "highest-scoring **in their league**" — which needs full per-user roster data (Phase 5 deliberately skipped importing this as speculative) cross-referenced across three providers with three different, never-reconciled player-ID schemes, and ultimately collides with DESIGN.md's own explicit non-goal that multi-game tracking is out of v1 scope. Spending budget here would optimize a heuristic that still wouldn't reach the stated goal. `checker/app.py`'s `_pick_game_to_track` docstring now carries this full reasoning inline rather than reading like an unfinished stub.
